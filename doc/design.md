# SmartMediaDisk 设计需求规格说明书

## 1. 文档目的
本文档用于定义 SmartMediaDisk 软件系统的设计需求、功能范围、技术约束与验收标准，作为后续开发、测试、部署与扩展的统一依据。

## 2. 项目概述
SmartMediaDisk 旨在基于用户现有硬件提供便捷、可靠、可扩展的文件管理与存取方案。系统以 Docker 容器化方式部署，确保跨平台可用，并降低运维复杂度。

## 3. 设计目标
- 提供统一、易用的文件上传、下载、组织、共享能力。
- 支持局域网与非局域网场景下的多渠道文件传输。
- 通过内容寻址（SHA256）与编号机制降低冗余存储占用。
- 提供细粒度权限模型，实现用户、用户组与权限点的灵活管理。
- 预留插件化接口，支持文件预览能力持续扩展。
- 面向不同使用场景提供差异化 UI（家庭媒体中心/企业数据中心）。

## 4. 系统范围
### 4.1 范围内
- 容器化部署架构设计与运行方案。
- 前后端与数据库基础能力建设。
- 文件传输、文件管理、用户与权限管理、文件预览、场景化 UI。
- 外部服务接口（邮件、第三方网盘）集成能力。

### 4.2 范围外
- 第三方网盘厂商私有协议的深度定制实现（仅实现开放接口对接层）。
- 专有商业预览引擎内部逻辑实现（仅支持接入）。

## 5. 技术架构与技术栈
### 5.1 总体架构
- 部署形态：Docker 容器。
- 运行模式：前后端分层 + 数据持久化。
- 扩展方式：模块化接口 + 插件机制。

### 5.2 技术栈约束
- 前端：由 Nginx 托管 HTML 页面与静态资源。
- 后端：Django 实现业务逻辑与 API。
- 数据库：SQLite 持久化存储元数据。
- 邮件能力：Sendmail 用于非局域网小文件传输与通知。

### 5.3 容器化要求
- 系统应可在主流平台（Linux/macOS/Windows）通过 Docker 启动。
- 应提供可复用的镜像构建与运行配置（如 Dockerfile、compose 配置）。
- 数据目录与配置目录应支持宿主机挂载，保障升级与重建容器后的数据可恢复。

系统后端以 Django API 为中心，Nginx 负责静态资源托管和 API 反向代理。模块关系如下：

```text
用户请求
  ↓
Nginx
  ├─ 静态页面：src/html
  └─ /api/ 反向代理到 Django
        ↓
Django API
  ├─ accounts：登录、会话、用户信息
  ├─ permissions：权限点、用户组、继承判断
  ├─ files：文件操作入口
  │    └─ storage：SHA256 去重、物理文件池、引用计数

  ├─ preview：文本/图片/音频/Office/PDF 预览
  ├─ transfer：上传下载任务、重试、状态
  ├─ mailer：Sendmail/SMTP 分发
  └─ audit：关键操作审计
```

## 6. 功能需求
### 6.1 文件传输
#### 6.1.1 局域网传输
- 用户可通过 Web 页面完成双向文件传输（上传/下载）。
- 用户可通过 FTP 等文件传输协议访问系统进行传输。
- 系统应记录基础传输日志（时间、用户、来源、目标、结果）。

#### 6.1.2 非局域网传输
- 支持邮件传输小型文件（通过 Sendmail）。
- 支持通过第三方网盘开放接口进行文件中转或直传。
- 应提供统一的任务状态反馈（排队、进行中、成功、失败）。

#### 6.1.3 传输约束
- 超过邮件大小阈值时，系统应提示改用第三方网盘通道。
- 对传输失败任务提供可重试机制与失败原因提示。

### 6.2 文件管理
#### 6.2.1 文件唯一标识
- 文件内容以 SHA256 作为主标识。
- 当 SHA256 冲突或存在业务区分需求时，引入“编号”作为次级区分字段。
- 文件逻辑 ID 建议采用 `sha256 + serial` 组合形式。

#### 6.2.2 去重与存储优化
- 相同内容文件在物理存储中仅保留一份。
- 多个逻辑文件记录可映射到同一物理文件。
- 应支持在文件管理器中构建软链接/硬链接关系（按平台能力适配）。

#### 6.2.3 元数据管理
- 使用 SQLite 存储文件元数据（名称、路径、哈希、大小、创建时间、归属用户、引用计数等）。
- 删除文件时应基于引用计数决定是否删除物理文件。

### 6.3 用户与权限管理
#### 6.3.1 权限模型
- 系统权限参考 LuckPerms 的节点化授权思路，所有权限均表达为层级式字符串节点，例如 `files.read.own`、`files.share.all`、`permissions.table.edit`。
- 权限节点使用英文小写、点号分隔的层级命名，格式为 `<domain>.<resource-or-scope>.<action>` 或 `<domain>.<action>.<scope>`，同一领域内保持稳定顺序。
- 权限支持通配符节点 `*` 和分段通配符，例如 `files.*`、`files.read.*`。通配符只匹配同层级及其子层级，不用于模糊文本搜索。
- 权限关系保存 `node`、`value`、`source`、`context`、`expires_at` 等字段：
  - `node`：权限节点字符串。
  - `value`：布尔值，`true` 表示允许，`false` 表示显式拒绝。
  - `source`：权限来源，可为用户直接授权、用户组继承或系统内置。
  - `context`：可选上下文，用于未来扩展路径、设备、访问来源等限制。
  - `expires_at`：可选过期时间，用于临时授权。
- 权限判定规则：
  - `root` 用户拥有系统内置的不可移除 `*` 权限，始终视为全权限。
  - 普通用户默认无任何权限；本地默认 `user` 用户初始权限为空。
  - 未命中任何有效允许节点时默认拒绝。
  - 更具体的节点优先于更宽泛的通配符，例如 `files.delete.own` 优先于 `files.*`。
  - 同等具体度下，用户直接授权优先于用户组继承；显式拒绝优先于允许。
  - 已过期权限不参与判定。

#### 6.3.2 用户组与继承
- 参考 LuckPerms 思路实现“用户-组-权限”三层关系。
- 用户加入用户组后自动继承组权限。
- 支持用户直接授权（用于特殊例外场景），并按“用户直接授权 > 用户组授权 > 默认策略”的顺序判定。
- 用户组可设置权重 `weight`，当多个组同时提供同等具体度的冲突节点时，权重更高的组优先。
- 首期实现允许先完成用户直接权限表编辑；用户组继承模型保留数据结构和判定接口，便于后续扩展批量授权。

#### 6.3.3 权限能力
- 支持权限新增、修改、回收、审计。
- 权限变更应具备日志记录与可追溯能力。
- 权限管理页仅允许具备“编辑权限表”权限的用户进入和保存变更，该权限节点定义为 `permissions.table.edit`。
- 前端页面入口、页面加载 API、保存 API 均必须校验 `permissions.table.edit`；仅隐藏菜单不能作为安全边界。
- 权限变更必须记录操作者、目标用户、节点、原值、新值、时间、来源 IP 或会话标识。
- `*` 节点只允许作为系统内置 root 权限存在，不允许在页面中授予、修改或删除。

#### 6.3.5 默认账号与初始权限
- `root`：默认密码 `123456`，系统内置 `*` 权限，不能通过权限管理页取消或降级；由于当前系统使用用户名 `root` 识别内置全权限，root 用户名不可修改。
- `user`：默认密码 `123456`，初始无权限，用于测试普通用户在无授权状态下的访问表现。
- 首期不为 `user` 自动分配默认组权限，避免测试权限差异时受到隐式授权影响。
- 权限管理页应明确标识 root 的 `*` 为系统锁定状态，保存时后端也必须忽略对该锁定权限的删除请求。

### 6.4 文件预览
- 对可预览文件类型（如图片、文本、PDF 等）提供在线预览入口。
- 预览能力支持两种接入方式：
  - 接入外部开源预览方案；
  - 提供标准插件接口由开发者自行扩展。
- 对不支持预览的类型应提供明确提示与下载选项。

### 6.5 用户界面
- 提供至少两套前端交互模式：
  - 家庭媒体中心模式：强调易用、直观、轻操作。
  - 企业数据中心模式：强调批量处理、权限可视化、任务监控。
- 不同模式共享同一后端能力，但在布局、导航、信息密度上可差异化配置。

### 6.6 低优先级功能（预研）
- 针对具备 AI 加速硬件（如 OrangePi）设备，支持文件内容分析能力（如内容摘要、分类标签、相似文件推荐）。
- 该能力默认关闭，仅在检测到可用硬件与模型依赖后启用。

## 7. 非功能性需求
### 7.1 可用性
- 常用操作（上传、下载、重命名、移动）应在 Web 端可直达。
- 关键任务应有明确状态与错误提示。

### 7.2 性能
- 局域网场景下应优先保障大文件传输稳定性。
- 去重与元数据查询应在可接受时延内完成（目标值后续按压测结果细化）。

### 7.3 可扩展性
- 文件预览、第三方网盘、权限策略应采用可插拔设计。
- 未来迁移数据库（如 PostgreSQL）时应尽量减少业务层改动。

### 7.4 安全性
- 所有文件访问请求必须经过权限校验。
- 对关键操作（删除、授权变更、外链传输）记录审计日志。
- 对上传内容进行基本安全检查（扩展名/类型校验、大小限制等）。

### 7.5 可维护性
- 模块边界清晰（传输、存储、权限、预览、界面）。
- 配置项集中管理，支持环境变量覆盖。

## 8. 数据设计（建议）
### 8.1 核心实体
- `User`：用户信息。
- `Group`：用户组。
- `Permission`：权限点定义。
- `UserGroupRelation`：用户与组关系。
- `GroupPermissionRelation`：组与权限关系。
- `UserPermissionRelation`：用户直接权限关系。
- `FileObject`：物理文件对象（按 `sha256` 管理）。
- `FileEntry`：逻辑文件记录（包含 `sha256 + serial`、路径、归属、引用关系）。
- `TransferTask`：文件传输任务记录（渠道、状态、重试次数、错误信息）。

### 8.2 关键约束
- `FileObject.sha256` 唯一。
- `FileEntry` 在相同 `sha256` 下 `serial` 唯一。
- 删除 `FileEntry` 时若引用计数为 0，才允许回收 `FileObject`。

### 8.3 核心数据模型职责
- `User`：使用 Django 内置用户模型作为账号基础，承载登录身份与后台管理能力。
- `Group`：用户组，用于批量承载权限配置。
- `PermissionNode`：权限点定义，例如 `permissions.table.edit`、`files.read.own`，记录显示名、分类、说明、是否危险操作、是否系统锁定。
- `UserGroupRelation`：用户与用户组关系。
- `GroupPermissionRelation`：用户组与权限点关系，记录 `node`、`value`、`context`、`expires_at`，用于组继承授权。
- `UserPermissionRelation`：用户直接权限关系，记录 `node`、`value`、`context`、`expires_at`，用于特殊例外场景；root 的 `*` 由系统保留记录或运行时规则提供，不允许页面删除。
- `FileObject`：物理文件对象，记录 `sha256`、文件大小、MIME 类型、存储路径和引用计数。
- `FileEntry`：逻辑文件记录，记录归属用户、文件名、逻辑路径、`sha256`、`serial` 和创建时间。
- `ShareLink`：分享链接，记录访问令牌、关联文件、分享权限、过期时间和访问范围。
- `TransferTask`：传输任务，记录任务渠道、状态、重试次数、错误信息和执行时间。
- `PreviewCache`：预览缓存，记录缩略图、转换后的 HTML/PDF、音频元信息等缓存资源。
- `AuditLog`：审计日志，记录删除、下载、授权、分享等关键操作。

### 8.4 权限数据约束
- `PermissionNode.node` 唯一，节点命名固定后不得随意重命名；如需废弃，应标记为 inactive 并保留历史审计可读性。
- `UserPermissionRelation` 在同一用户、同一节点、同一上下文下唯一。
- `GroupPermissionRelation` 在同一组、同一节点、同一上下文下唯一。
- `root` 的 `*` 权限应通过系统锁定字段或鉴权服务硬编码保护，任何 API 都不得删除、置为 false 或设置过期时间。
- 对用户权限的批量保存应使用事务，避免部分节点写入成功导致权限表处于中间状态。

## 9. 接口设计要求（概要）
- 后端应提供 RESTful API 或等效接口供前端与外部模块调用。
- API 需统一认证、鉴权、错误码与分页规范。
- 外部传输适配层（邮件/网盘）应抽象统一接口，屏蔽渠道差异。

### 9.1 API 分组
- `/api/auth/`：登录、退出、会话状态。
- `/api/users/`：用户管理。
- `/api/groups/`：用户组管理。
- `/api/permissions/`：权限点管理。
- `/api/files/`：文件列表、上传、下载、移动、删除、重命名。
- `/api/storage/`：去重信息、引用计数、物理文件状态。
- `/api/shares/`：创建分享、访问分享、撤销分享。
- `/api/search/`：文件搜索、筛选。
- `/api/preview/`：文件预览、缩略图、转换状态。
- `/api/transfers/`：传输任务、重试、失败原因。
- `/api/mail/`：Sendmail/SMTP 文件分发。
- `/api/audit/`：审计日志。
- `/api/system/health/`：健康检查。

### 9.2 权限管理页面与接口
- 新增权限管理页面，建议路径为 `/permissions.html`，导航入口仅对拥有 `permissions.table.edit` 的用户显示。
- 页面加载时必须调用会话状态接口确认当前用户身份，再调用权限校验接口确认 `permissions.table.edit`；无权限时返回 403 并跳转或显示拒绝访问状态。
- 权限管理页面首期面向“用户直接权限”编辑：左侧列出用户，右侧按分类列出权限节点，支持勾选允许、显式拒绝、清空继承/直接配置。
- root 用户的 `*` 权限显示为锁定状态；页面不得提供取消入口，后端也必须拒绝任何修改 root `*` 的请求。
- `user` 用户初始权限为空，页面中应显示为未勾选状态，便于测试授权前后的差异。
- 建议接口：
  - `GET /api/permissions/nodes/`：返回可编辑权限节点定义，不包含可授予的 `*`。
  - `GET /api/permissions/users/`：返回用户列表及每个用户的有效权限摘要。
  - `GET /api/permissions/users/{id}/`：返回指定用户的直接权限、继承权限、最终权限和锁定权限。
  - `PUT /api/permissions/users/{id}/`：批量保存指定用户的直接权限；需要 `permissions.table.edit`。
  - `GET /api/permissions/check/?node=permissions.table.edit`：返回当前用户是否拥有指定权限，供前端控制页面入口。
  - `GET /api/permissions/audit/`：返回权限变更审计；需要 `permissions.audit.view`。
- 所有权限写接口都必须记录审计日志，日志至少包含操作者、目标用户、节点、旧值、新值和操作时间。

## 10. 部署与运维要求
- 提供一键启动方式（如 `docker compose up`）。
- 支持日志分级与按模块输出。
- 关键目录（数据、配置、日志）支持挂载与备份。
- 应提供基础健康检查接口用于容器编排探活。

## 11. 测试与验收标准
### 11.1 功能验收
- 文件上传、下载、移动、删除、重命名功能可用。
- 文件去重机制生效（相同内容仅一份物理存储）。
- 局域网（Web/FTP）与非局域网（邮件/网盘）传输链路可用。
- 用户组权限继承与权限点校验结果符合预期。
- 可预览类型文件可正常预览，不可预览类型提示正确。

### 11.2 稳定性验收
- 容器重启后数据可恢复。
- 异常传输任务可追踪并可重试。

### 11.3 安全验收
- 未授权用户不可访问受限文件。
- 关键操作具备可审计记录。

## 12. 里程碑建议
- M1：基础架构与容器化（Nginx + Django + SQLite + Sendmail）打通。
- M2：文件传输与文件管理核心能力（去重 + 元数据）上线。
- M3：用户组权限系统上线（权限点 + 继承 + 审计）。
- M4：文件预览插件机制与双场景 UI 初版。
- M5（低优先级）：AI 硬件适配能力预研与灰度。
